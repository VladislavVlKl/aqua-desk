📋 Что нужно для мерчанта

Минимум — ИП: регистрация через my.gov.uz, 1–3 дня, минимальная госпошлина. Достаточно для оплаты абонементов внутри одного клуба.

Для SaaS — ООО: выглядит серьёзнее, проще заключать договоры с другими клубами, привлекать соучредителей.

Лицензия не нужна. Лицензируются только операторы платёжных систем (банки). Вы — мерчант. Основание: Закон РУз «О платёжах и платёжных системах» №ЗРУ-598 от 2019 г., ст. 5.

🧾 Онлайн-ККМ — обязателен по закону

Да, обязателен. Налоговый кодекс РУз, ст. 272 + Постановление Президента №ПП-4611 от 2020 г. Касается всех безналичных расчётов с физлицами, включая Payme/Click.

Как работает: Payme и Click имеют встроенную интеграцию с ОФД — при правильной настройке мерчанта чек формируется автоматически. Нужно зарегистрировать виртуальный ККМ в Налоговом комитете и подключить к ОФД.

Штрафы за отсутствие: 5–30 БРВ (1,7–10 млн сум) по ст. 175 КоАО РУз. Налоговые органы делают контрольные закупки.

⚡ Техническая схема для AquaDesk
// Схема оплаты через Supabase Edge Function Mini App → Telegram.WebApp.openLink(payme_checkout_url) → Пользователь оплачивает в Payme → Payme callback → Supabase Edge Function (публичный HTTPS URL) → Edge Function обновляет баланс клиента в БД → Mini App polling обнаруживает обновление

UX: пользователь уходит из Telegram в приложение Payme, потом возвращается. Нативного платёжного экрана внутри Mini App нет — это нормально для рынка.

Секреты мерчанта должны быть только в Edge Functions, не во фронтовом JS!

💳 Расчёты в долларах (SaaS)

Важно: расчёты между юрлицами Узбекистана в иностранной валюте запрещены (Закон «О валютном регулировании» ЗРУ-561). Если клиент — узбекское ООО/ИП, счёт нужно выставлять в сумах по курсу ЦБ.

Для зарубежных клиентов — нужен валютный счёт и обязательная продажа части валютной выручки.

Если монетизация маловероятна → HR-инструмент Для внутреннего использования без приёма внешних платежей — ничего из вышеперечисленного не нужно. Работайте как есть, никаких юридических рисков.

Шаг 2 — Аудит безопасности
Найдено: 4 критических + 5 средних уязвимостей
Критично: dev-режим активен в продакшне Любой человек, открывший приложение вне Telegram, видит prompt() с запросом ID — и может ввести чужой. Закрывается за 10 минут.
КРИТИЧНО Dev-режим активен в продакшне (app.js:146–153)

Fallback с prompt('Dev mode: Telegram ID') срабатывает когда приложение открыто вне Telegram. GitHub Pages доступен в любом браузере — любой может ввести чужой tg_id и войти под его профилем.

// Исправление — 3 строки: STATE.tgId = window.Telegram?.WebApp?.initDataUnsafe?.user?.id || null; if (!STATE.tgId) { setScreen('<div class="center-screen"><p>Откройте приложение через Telegram</p></div>'); return; }

Время исправления: 10 минут.

КРИТИЧНО Авторизация по tg_id без верификации initData

Приложение доверяет initDataUnsafe без проверки HMAC-SHA256 подписи Telegram. Знающий tg_id другого сотрудника может обратиться к Supabase API напрямую от его имени — если RLS это не блокирует.

Решение: Supabase Edge Function, которая верифицирует initData через bot token и выдаёт JWT. Требует ~1 день работы. Для 20 внутренних пользователей — приоритет средний.

КРИТИЧНО Проверка ролей только в JS (не в базе)

Разграничение доступа (renderAdminApp, renderTrainerApp) происходит только в браузере. Тренер, зная Supabase URL и anon key, может вызвать getAllClients, getSummary, getControlData напрямую и получить данные всех коллег и зарплаты.

Решение: Проверить RLS-политики в Supabase Dashboard. Они должны ограничивать доступ на уровне базы, а не только в коде.

КРИТИЧНО Brute-force PIN — счётчик сбрасывается при перезагрузке

Переменные _pinFailCount и _pinBlockedUntil живут только в памяти браузера. Обновление страницы = сброс счётчика. 4-значный PIN можно перебрать за 10 000 попыток.

-- Добавить в RPC verify_pin (PostgreSQL): IF (SELECT COUNT(*) FROM pin_attempts WHERE tg_id = p_tg_id AND attempt_at > NOW() - INTERVAL '5 minutes' ) >= 5 THEN RETURN FALSE; END IF;
СРЕДНЕ anon key в публичном репозитории GitHub

Это ожидаемо для Supabase — anon key предназначен для клиентов. Критичным становится только если RLS настроен неверно. Проверьте RLS — это важнее, чем скрывать ключ.

СРЕДНЕ XSS через innerHTML с данными из БД

ФИО клиентов, названия групп, заметки вставляются в HTML через шаблонные строки без экранирования. Если кто-то введёт <script>...</script> в поле ФИО — JS выполнится у других пользователей. Для внутренней системы с доверенными пользователями — низкий реальный риск.

✅ Что сделано правильно

  • PIN хранится хешированным, проверяется через RPC — не виден клиенту ✓
  • Защита от двойных нажатий (once(), _pending) — нет дублей тренировок ✓
  • Атомарное списание баланса через RPC increment_balance ✓
  • Архивирование вместо удаления — история сохраняется ✓
  • Лимит 48ч с аудит-треком поздних запросов ✓
После внедрения защиты — разработка не усложнится Все существующие DB.* функции остаются без изменений. Только инициализация клиента Supabase изменится (JWT вместо anon key напрямую). RLS работает прозрачно.

Шаг 3 — Функциональный аудит
Слабые места + объяснение про "5000 строк без тестов"

💡 Что значит "5000 строк без тестов" — простым языком

Представьте сложную машину из 5000 деталей. Тесты — это автопроверки: нажал кнопку, машина сама прогнала 50 сценариев и говорит "всё ОК" или "вот здесь сломалось". В AquaDesk таких проверок нет — каждое изменение нужно проверять руками.

Опасно ли это при 20 пользователях? Умеренно. Реальный риск не в падении системы, а в тихих ошибках: тренер отработал, но в ЗП не попало; баланс клиента стал 0 хотя ПТ ещё есть. Такое может долго не замечаться.

Нужно ли исправлять сейчас? Нет. При 20 внутренних пользователях правильнее исправить конкретные баги (таблица ниже), а не писать тесты. Тесты понадобятся при 100+ пользователях или втором разработчике.

Использовать только внутри своего клуба, без продаж

→ Первое действие: замерить реальную экономию времени за 2–3 месяца. Это станет главным аргументом при продаже другим клубам.

🚀 5 функций которые максимально увеличат готовность платить

  1. Автоматические уведомления клиентам — напоминание об истекающем абонементе в Telegram клиента. ROI виден сразу: "напомнил 10, 7 продлили".
  2. Касса / учёт оплат наличными — Узбекистан наличный рынок. Журнал: кто, сколько, за что. Убийца Excel-таблицы у администратора.
  3. Дашборд выручки для владельца — простой экран: сколько заработал клуб, какой тренер приносит больше, заполненность групп. Владелец платит — ему нужны цифры.
  4. Онлайн-запись клиентов через Telegram — клиент сам записывается к тренеру. Снимает нагрузку с администратора, выглядит современно.
  5. Фото прогресса клиента — тренер загружает фото раз в месяц, клиент видит результат. Инструмент удержания клиентов = аргумент для владельца клуба.

📋 Рекомендуемая последовательность

0–3 мес: HR-инструмент для себя + добавить кассу наличных + дашборд владельца. Замерить экономию.

3–9 мес: 3–5 клубов по модели "установка под ключ" ($200–300 + $40/мес поддержка). Реальные кейсы и отзывы.

9–24 мес: при 15–20 клубах — инвестировать в мультиарендность и переходить к SaaS.

Главный вывод: Узбекский рынок покупает через доверие и личный контакт. Не пытайтесь сразу строить SaaS. Белый лейбл — и быстрые деньги, и путь к пониманию рынка.
