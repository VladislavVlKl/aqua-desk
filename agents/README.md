# AquaDesk — Мультиагентная система

Четыре агента на Claude API: оркестратор и три специалиста. Ты общаешься только с оркестратором, он маршрутизирует задачи и собирает результат.

```
Ты
 │
 ▼
Оркестратор (orchestrator.md)
 ├──► Архитектор кода (code-architect.md)
 ├──► Архитектор БД   (db-architect.md)
 ├──► DevOps          (devops.md)
 └──► Дефендер        (defender.md)  — аудитор безопасности, фиксы отдаёт через HANDOFF
```

## Файлы

| Файл | Агент | Модель (рекомендация) |
|---|---|---|
| `orchestrator.md` | Оркестратор | `claude-fable-5` |
| `code-architect.md` | Архитектор кода | `claude-fable-5` |
| `db-architect.md` | Архитектор БД | `claude-fable-5` |
| `devops.md` | DevOps | `claude-sonnet-4-6` |
| `defender.md` | Дефендер (security) | `claude-opus-4-8` |

Каждый `.md` — готовый системный промпт: содержимое файла целиком передаётся в параметр `system` при вызове API.

## Ключевые принципы (зашиты в промпты)

1. **Исполнитель со шлюзом одобрения** — агент анализирует и готовит изменения сам, но ничего не применяет без явного «да» от пользователя. Формат шлюза описан в каждом промпте.
2. **Контекст-якоря** — каждому агенту в первое сообщение подкладываются актуальные `CLAUDE.md` + его профильный документ (`ARCHITECTURE.md` или `DATABASE.md`). Это снижает риск решений «по учебнику», игнорирующих ограничения проекта.
3. **Протокол передачи между агентами** — структурированный JSON-блок `HANDOFF` (см. ниже). Оркестратор пересылает его целиком, ничего не пересказывая своими словами.
4. **Чеклисты перед шлюзом** — у архитектора БД и DevOps есть обязательные чеклисты; пропуск пункта = нельзя выходить на шлюз.

## Протокол HANDOFF (между агентами)

Когда агент передаёт задачу другому, он завершает ответ блоком:

```json
{
  "handoff_to": "db-architect | code-architect | devops",
  "task": "одно предложение — что нужно сделать",
  "context": "факты, которые принимающий агент не может узнать сам",
  "artifacts": ["пути к файлам / SQL / diff"],
  "blocking": true
}
```

`blocking: true` — оркестратор ждёт результата перед продолжением. Оркестратор передаёт блок принимающему агенту без изменений.

## Протокол шлюза одобрения (агент → пользователь)

Перед любым изменением (запись файла, миграция, push) агент выводит:

```
═══ ШЛЮЗ ОДОБРЕНИЯ ═══
Что: <одно предложение>
Зачем: <одно предложение>
Изменения: <diff / SQL / команды — полностью, без сокращений>
Откат: <как отменить, если что-то пойдёт не так>
Риски: <что может сломаться; «нет» — если действительно нет>
══════════════════════
Жду: да / нет / правки
```

Применять можно только после явного «да» в следующем сообщении пользователя. Молчание, смена темы, уточняющий вопрос — НЕ одобрение.

## Минимальный каркас вызова (Python)

```python
import anthropic, pathlib

client = anthropic.Anthropic()  # ANTHROPIC_API_KEY в окружении
AGENTS = pathlib.Path(__file__).parent

def ask(agent: str, messages: list[dict]) -> str:
    system_prompt = (AGENTS / f"{agent}.md").read_text()
    # контекст-якоря проекта
    root = AGENTS.parent
    anchors = {"orchestrator": ["CLAUDE.md"],
               "code-architect": ["CLAUDE.md", "ARCHITECTURE.md"],
               "db-architect": ["CLAUDE.md", "DATABASE.md"],
               "devops": ["CLAUDE.md"],
               "defender": ["CLAUDE.md", "DATABASE.md", "ARCHITECTURE.md"]}
    ctx = "\n\n".join(f"<doc name=\"{f}\">\n{(root/f).read_text()}\n</doc>"
                      for f in anchors[agent] if (root/f).exists())
    # Системный блок статичен в пределах диалога → кешируем его.
    # cache_control на последнем блоке кеширует весь префикс system.
    # Кеш живёт ~5 мин, бьётся при правке .md или якоря; messages НЕ кешируем
    # (они меняются каждый ход и сбили бы кеш).
    system = [
        {"type": "text", "text": system_prompt},
        {"type": "text",
         "text": "# Актуальные документы проекта\n" + ctx,
         "cache_control": {"type": "ephemeral"}},
    ]
    resp = client.messages.create(
        model="claude-fable-5",
        max_tokens=8192,
        system=system,
        messages=messages,
    )
    return resp.content[0].text
```

> **Caching (п.1 оптимизации токенов):** системный блок (промпт агента + якоря) одинаков на каждом ходе диалога, поэтому помечается `cache_control: ephemeral` — повторные ходы читают его из кеша по ~0.1× цены. Порядок блоков не менять и волатильный контент (сообщения пользователя) внутрь кешируемого префикса не класть, иначе кеш не попадёт.

Инструменты (чтение/запись файлов, git, Supabase) подключаются через tool use — схемы инструментов перечислены в промпте каждого агента в секции «Инструменты». На первом этапе можно запускать агентов без инструментов, в режиме «советника»: они выдают diff/SQL/команды, а применяешь ты руками или через Claude Code.

## Порядок внедрения

1. **Этап 1 — советники.** Гонять промпты без tool use, изменения применять самому. Цель: проверить качество решений и откалибровать промпты.
2. **Этап 2 — исполнители со шлюзом.** Подключить инструменты (файлы — архитектору кода, Supabase MCP — архитектору БД, git — DevOps). Шлюз одобрения обязателен.
3. **Этап 3 — связки.** Включить HANDOFF-протокол через оркестратора на реальной задаче из беклога (хороший кандидат — задача #6: восстановление ПТ-клиента из архива, она затрагивает и БД, и код).
